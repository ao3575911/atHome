# v0.3 Hosted Registry Architecture Spec

## Goal

Define the minimum hosted registry architecture needed for v0.3 alpha while preserving independent verification of identity manifests, revocations, signed requests, and witness receipts.

## Non-goals

- Replacing local-first operation.
- Custodying production private keys by default.
- Building a global consensus network in v0.3.

## Core Components

| Component       | Responsibility                                                                              |
| --------------- | ------------------------------------------------------------------------------------------- |
| Registry API    | HTTP interface for identity, namespace, service, agent, token, and revocation operations    |
| Storage adapter | Durable backend abstraction for manifests, events, revocations, replay state, and witnesses |
| Event log       | Append-only signed mutation history                                                         |
| Witness service | Signs receipts/checkpoints over registry events                                             |
| Resolver        | Verifies manifests and resolves root/service/agent names                                    |
| Ops console     | Reviews health, abuse, audit, namespace, and user states                                    |

## Storage Model

Required logical tables/collections:

- `namespaces`
- `identity_manifests`
- `public_keys`
- `services`
- `agents`
- `capability_tokens`
- `revocations`
- `registry_events`
- `witness_receipts`
- `replay_nonces`
- `abuse_reviews`

The existing local JSON backend should remain as the development adapter. v0.3 should introduce one SQL adapter behind the same protocol-level interfaces.

## Write Path

1. Client submits mutation with `X-Home-Authorization`.
2. API validates request schema.
3. API verifies root mutation signature over method/path/body.
4. Policy engine checks namespace/key/agent state.
5. Registry applies mutation transactionally.
6. Registry emits signed event.
7. Witness service signs receipt/checkpoint.
8. API returns public state and receipt metadata.

## Read Path

1. Client requests manifest, resolve, revocation, or witness data.
2. API returns public data plus freshness metadata.
3. Client or SDK verifies canonical signature locally.
4. Resolver rejects stale or unverifiable state when configured to require freshness.

## Freshness Contract

v0.3 should expose:

- manifest `updatedAt`
- registry checkpoint id
- witness receipt id
- revocation index timestamp
- max accepted staleness config for SDK verification

## Abuse and Suspension

Namespace suspension must be represented as signed registry state, not only an internal database flag. Clients need to distinguish:

- active
- suspended
- revoked
- transferred
- recovery-pending

## Postgres Migration Approach

`PostgresRegistryBackend` is the v0.3 SQL adapter. It implements the full `RegistryBackend` interface (same contract as `LocalJsonStore` and `SQLiteRegistryBackend`) and loads `pg` lazily via `new Function("return import('pg')")()` so it remains an optional dependency — not installed unless needed.

SQL DDL lives in `packages/protocol/src/storage/migrations/001_initial.sql`. It covers all 14 protocol tables and is idempotent (`CREATE TABLE IF NOT EXISTS`). Run it via:

```typescript
const backend = createPostgresRegistryBackend({
  connectionString: DATABASE_URL,
});
await backend.runMigrations();
```

Or use the built-in verification script:

```bash
DATABASE_URL=postgres://user:pass@host/db npm run verify:postgres
```

That script runs `runMigrations()`, then executes the full `backend.test.ts` parity suite against the live database.

### Backend Parity Suite

`packages/protocol/test/backend.test.ts` uses a shared `runBackendParitySuite(label, makeBackend)` helper that runs the same 7 test cases against every adapter:

- memory backend (always)
- `LocalJsonStore` backend (always)
- `SQLite` backend (always)
- Postgres backend (skipped when `DATABASE_URL` is absent)

This ensures all adapters satisfy the same event-append, hash-chain, signature, witness-receipt, checkpoint, custody-key, and replay-nonce contracts.

## Acceptance Criteria

- Storage adapter interface exists and local JSON still passes all current tests. ✅
- Hosted storage implementation (`PostgresRegistryBackend`) has integration tests via parity suite. ✅
- Resolver can report freshness metadata. ✅
- Registry events are generated for namespace, key, service, agent, token, and revocation mutations. ✅
- Docs include hosted deployment topology and failure modes. ✅
