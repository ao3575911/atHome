# v0.3 Production Key Custody Spec

## Goal

Move atHome from demo private-key export toward explicit production custody modes where private keys are never accidentally returned by hosted API responses.

## Custody Modes

### Local development

- Ed25519 keys may be generated and exported only when `ATHOME_DEMO_PRIVATE_KEY_EXPORT=true`.
- This mode is rejected when `NODE_ENV=production`.
- Generated demo key material stays gitignored runtime data.

### Browser-held keys

- Browser creates or imports a signing key.
- API receives only public key material and signed mutation headers.
- SDK provides WebCrypto-compatible signing helpers where possible.

### Passkey/WebAuthn-assisted authorization

- User signs authorization ceremonies using passkeys.
- Root identity operations may require WebAuthn assertion plus protocol signature.
- Recovery methods should be explicit manifest entries.

### Hosted operator custody

- Optional service-operator custody via KMS/HSM.
- Keys are referenced by id and purpose.
- Signing operations are audited.
- Raw private keys are never exported through API responses.

## API Rules

- Production bootstrap endpoints must not return private keys.
- Mutation routes must verify root authorization headers.
- Error details must not include private key material, raw signing payload secrets, or replay nonce internals.
- Key creation responses should return public material, key id, custody mode, and recovery guidance.

## SDK External Signing Boundary

`createExternalMutationSigner(...)` lets clients authorize mutations with an async signing boundary instead of passing raw private keys into the SDK. The SDK constructs the canonical mutation draft, then delegates only the draft signature operation to the caller-provided signer. This supports browser-held keys, passkey-mediated ceremonies, KMS/HSM signing services, and future custody adapters.

## Root Key Rotation

Root key rotation is exposed through `POST /identities/:id/keys/root/rotate`. The route requires mutation authorization from the currently active root key, deprecates the old root key, creates a new active root key, re-signs the manifest with the new root key, and returns custody metadata. New private key material is only returned when local demo export is explicitly enabled.

## Recovery Lifecycle

1. Register recovery method.
2. Verify recovery method.
3. Start recovery ceremony.
4. Apply waiting period / witness checkpoint.
5. Rotate or restore root key.
6. Emit signed registry event.
7. Notify configured service endpoints.

## Threat Model Notes

Primary risks:

- server-side private-key leakage
- replayed mutation authorization
- compromised delegated agent key
- stale revocation state
- malicious namespace transfer
- user lockout after root-key loss

## Acceptance Criteria

- Production mode test proves private keys are absent from API responses.
- SDK exposes custody-mode-aware helpers, including `createExternalMutationSigner(...)` for browser, passkey, KMS, HSM, or other external signing boundaries that should not hand raw private keys to the SDK.
- Docs explain how to run local demo mode without implying production safety.
- Key rotation and revocation flows are covered by tests and examples.
