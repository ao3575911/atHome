# atHome v0.3 Build Plan / Spec

## Objective

Turn the v0.2 local-first developer implementation into a v0.3 alpha that has a credible hosted-registry path, stronger security posture, CI-backed repository operations, and web/API surfaces ready for real integration work.

## Current Baseline

- `main` and `master` are reconciled to the same repository state.
- API, protocol package, SDK, and Next.js web platform live in one npm run monorepo.
- CI verifies typecheck, web typecheck, Vitest, Prettier, and production web build.
- Discussions, issue templates, labels, and security policy are enabled.
- Demo private-key runtime data is no longer tracked in the active tree.

## Build Principles

1. **Local-first remains real.** Hosted services must not remove the ability to verify manifests, signatures, revocations, and requests independently.
2. **Private keys do not become server defaults.** Demo export paths stay explicitly local/dev only; production signing gets a separate custody boundary.
3. **Every mutation is auditable.** Registry writes, revocations, namespace changes, and key lifecycle events need signed event trails.
4. **Web surfaces must reflect protocol state.** Mock panels are acceptable for scaffolding only; v0.3 should wire core read flows to real API responses.
5. **CI gates merge quality.** `main` stays releasable.

## Phase 1 — Repository and Security Hardening

### 1.1 Close branch-history cleanup

- Confirm `main` and `master` heads match.
- Close issue #7 with the reconciled commit hash.
- Keep `main` as the default branch.

### 1.2 Branch protection

- Require pull requests before merging to `main`.
- Require CI `verify` to pass.
- Keep delete-branch-on-merge enabled.
- Prefer squash merges for feature branches.

### 1.3 Demo-key history cleanup

- Treat all previously published demo private keys as compromised samples.
- Regenerate demo identities, manifests, revocations, and events if fixtures need to stay meaningful.
- Decide whether history rewrite is worth the disruption; if not, document the compromise boundary clearly.
- Never reuse historical demo private keys for future fixtures.

## Phase 2 — Hosted Registry Spec

### 2.1 Persistence boundary

- Add a storage interface suitable for Postgres/SQLite while preserving the existing local JSON adapter.
- Define migrations for identities, manifests, keys, services, agents, revocations, events, and witness receipts.
- Keep canonical payloads intact so external verification remains stable.

### 2.2 Namespace lifecycle

- Reserve namespace.
- Create root identity.
- Register service endpoints.
- Register delegated agents.
- Transfer namespace ownership.
- Recover root identity.
- Suspend/restore abusive or compromised namespaces.

### 2.3 Consistency and freshness

- Define freshness windows for manifest reads and revocation checks.
- Add signed registry checkpoints.
- Add witness receipt verification.
- Expose replication status for ops and clients.

## Phase 3 — Production Key Custody

### 3.1 Custody modes

- Local/dev exported Ed25519 keys.
- Browser-held keys with WebCrypto.
- Passkey/WebAuthn-backed authorization.
- KMS/HSM-backed hosted custody for service operators.

### 3.2 Key lifecycle

- Create root key.
- Add signing key.
- Rotate root key.
- Revoke key.
- Register recovery method.
- Execute recovery ceremony.

### 3.3 API contract changes

- Production responses must not return private keys.
- Bootstrap endpoints remain disabled in production.
- Mutation authorization errors must identify the failed verification boundary without leaking secrets.

## Phase 4 — API and SDK Build

### 4.1 API

- Add rate limits for bootstrap, mutation, resolve, and verify routes.
- Add abuse-review hooks for suspicious namespace/agent behavior.
- Add structured audit export endpoint.
- Add health/readiness/dependency status split.

### 4.2 SDK

- Add typed API error classes.
- Add browser/external-safe signing helpers that support async signing boundaries without exposing raw private keys to the SDK.
- Add capability/request verification helpers.
- Add namespace lifecycle client methods.
- Add examples for Node and browser runtimes.

### 4.3 OpenAPI

- Pin stable schema names.
- Add endpoint examples.
- Add auth-header examples.
- Add standard error-response examples.

## Phase 5 — Web Platform Build

### 5.1 Public site

- Wire namespace lookup to `/resolve`.
- Surface protocol status from API health/readiness.
- Add docs navigation for registry, custody, revocation, and SDK usage.

### 5.2 Developer portal

- Add local API connection settings.
- Add signed mutation playground.
- Add key-management and token-verification demos.
- Add SDK copy-paste snippets generated from real examples.

### 5.3 Ops console

- Wire namespace, audit, abuse, and health panels to real or fixture-backed API contracts.
- Add masked sensitive fields by default.
- Add event timeline views for registry mutations.

## Phase 6 — Release

### 6.1 Alpha release gates

- CI green on `main`.
- Security notes current.
- Demo fixtures regenerated or clearly marked compromised historical samples.
- Hosted registry and custody specs merged.
- Web/API integration path documented.

### 6.2 Release artifacts

- Tag `v0.3.0-alpha`.
- Publish release notes.
- Add screenshots or short demo video.
- Update Discussions roadmap with v0.4 candidates.
