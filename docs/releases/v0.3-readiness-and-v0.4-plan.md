# atHome Release Readiness Assessment and v0.4 Planning

Date: 2026-05-14 (UTC)  
Repo: https://github.com/ao3575911/atHome  
Assessed branch: `docs/realign-alpha2` (local checkout), plus live GitHub default branch and issue/PR state

## Executive Summary

`v0.3.0` stable is **not releasable** yet. `v0.3.0-alpha2` is a healthy alpha protocol/dev milestone, but production-hosted registry and production key custody are still incomplete and explicitly tracked as open blockers (#10, #11, #12, #21).

The most important distinction is still valid:

- **Alpha protocol/dev readiness:** mostly achieved.
- **Production hosted registry/custody readiness:** not achieved.

Recommended release posture: remain pre-release and avoid tagging `v0.3.0` stable.

## v0.3 Readiness Verdict

### Done

- Alpha tag exists: `v0.3.0-alpha2` published as prerelease (GitHub Releases).
- Branch protection and CI gates exist on `main`:
  - required status check: `verify`
  - PR review required
  - squash merge enabled, merge/rebase disabled, delete branch on merge enabled
- Core local/dev quality gates are largely in place and pass on CI for PR #20:
  - typecheck
  - web typecheck
  - tests
  - lint
  - web build
- Security posture improved versus earlier alpha notes:
  - high-severity fastify/fast-uri path called out as resolved in `SECURITY.md`
  - current `npm audit --audit-level=moderate` shows 7 moderate advisories

### Not Done

- Stable release blockers remain open and unchecked:
  - #10 durable hosted registry + namespace lifecycle
  - #11 passkey/WebAuthn + production custody path
  - #12 signed mutation playground + ops auth + full web/API integration
  - #21 stable release gate checklist
- Postgres backend is still a placeholder contract, not a working adapter.
- Namespace lifecycle endpoints are not present (`/namespaces/reserve|transfer|recover|suspend|restore` not implemented).
- Namespace lifecycle route surface is now present in alpha scaffolding:
  - `/namespaces/reserve`
  - `/namespaces/:id/suspend`
  - `/namespaces/:id/restore`
  - `/namespaces/:id/transfer`
  - `/namespaces/:id/recover`
  - Durable hosted-state semantics remain incomplete until Postgres-backed lifecycle state/migrations are implemented.
- Recovery-ceremony endpoints from #11 are not present.
- Web platform still has significant fixture/mock fallback in developer/ops panels.
- API/web version surfaces are inconsistent with alpha2 labeling.

### Blockers

1. **#10** Durable storage and namespace lifecycle (hard blocker for stable hosted deployment).
2. **#11** Production key custody with passkey/WebAuthn and KMS/HSM contract (hard blocker for stable security posture).
3. **#12** Real signed mutation UX + ops auth + live ops views (hard blocker for hosted operator workflows).
4. **#21** Stable release gate checklist remains open and not satisfied.

### Risks

- **Release-label confusion risk:** docs and runtime version strings are not fully aligned (examples below), increasing operator/user ambiguity.
- **Security/compliance drift risk:** accepted moderate advisories still unresolved; acceptable for alpha, not ideal for stable.
- **Implementation-doc drift risk:** issue bodies still reference old release-gates link (#13 PR) instead of #21 issue.

### Recommended Release Label / Version

- **Do not cut `v0.3.0` stable.**
- Keep pre-release line:
  - If shipping incremental hardening/docs only: `v0.3.0-alpha3`
  - If opening the next structured execution train: `v0.4.0-alpha1`

Given scope and remaining architecture/security work, `v0.4.0-alpha1` is the cleaner planning label.

## v0.4 Prioritized Roadmap

### Workstream 1 (P0): Durable hosted registry backend

- Implement real Postgres adapter with migration set for required tables.
- Add adapter parity tests (memory/SQLite/Postgres) in one shared suite.
- Implement namespace lifecycle mutations and event emission:
  - reserve, transfer, recover, suspend, restore
- Exit criteria:
  - Postgres parity green
  - namespace lifecycle endpoints + tests green
  - documented migration/runbook

### Workstream 2 (P0): Production key custody

- Implement passkey/WebAuthn custody provider and SDK signer helper.
- Define and stub KMS/HSM adapter contract (no private-key export path).
- Add recovery-method registration + recovery ceremony endpoints.
- Enforce production custody mode constraints in tests and runtime guards.
- Exit criteria:
  - passkey flow tested end-to-end (local demo harness)
  - production negative tests for key export pass

### Workstream 3 (P1): Security hardening and governance

- Add route-level admin auth for ops/destructive actions.
- Add abuse-control and rate-limit policy refinements.
- Resolve or bounded-mitigate moderate advisory set.
- Keep branch protections strict and codify release-safe merge policy.
- Exit criteria:
  - security checklist in #21 complete
  - dependency policy documented and enforced in CI

### Workstream 4 (P1): Real web/API integration

- Upgrade playground to perform real signed mutations (not resolve-only).
- Wire ops panels to live API for audit/registry/freshness/witness views.
- Keep explicit demo fallback labeling when API unavailable.
- Exit criteria:
  - local e2e smoke: create identity, signed mutation, audit visibility
  - no silent mock masquerading as live data

### Workstream 5 (P2): SDK and DX polish

- Expand typed error coverage and examples for browser signers.
- Add OpenAPI examples for auth/mutation/error envelopes.
- Publish “10-minute e2e quickstart” across API + SDK + web.
- Exit criteria:
  - copy-paste quickstart verified from clean clone
  - SDK examples cover external signer + browser signer paths
